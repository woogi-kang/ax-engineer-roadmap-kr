# 프로젝트 운영

## 현재 운영 방식

`v0.x` 기간에는 저장소 소유자가 maintainer 역할을 맡는다. 내용이 안정되고 반복 기여자가 생기면 reviewer와 영역별 maintainer를 추가한다.

## 결정 기준

로드맵 변경은 다음 순서로 판단한다.

1. 독자와 업무 결과에 어떤 문제가 있는가?
2. 현재 내용이 어떤 근거에 기반하는가?
3. 새 근거가 기존 판단을 어떻게 바꾸는가?
4. 특정 회사·도구에만 해당하는가, 공통 역량인가?
5. 학습·실습·증거 항목에 어떤 변화가 필요한가?

논쟁이 해결되지 않으면 하나의 정답을 강제하지 않고 조건별 선택지와 미확인 항목을 남길 수 있다.

## 버전

- Patch: 링크·오탈자·설명 보완
- Minor: 새 역량·사례·템플릿, 호환되는 구조 변경
- Major: 숙련도·생애주기·조직 성숙도처럼 핵심 계약을 바꾸는 변경

## 검토 주기

- 공식 직무 정의와 핵심 외부 링크: 분기별
- 기술·보안·규제 관련 내용: 변경 신호가 있을 때
- 사례와 템플릿: 사용·기여 피드백이 들어올 때
- 전체 정보 구조: Minor 릴리스 전

## 공개 범위

개인·회사·고객의 비공개 정보가 포함된 사례는 병합하지 않는다. 익명화만으로 재식별 위험을 제거할 수 없으면 구조와 판단만 일반화한다.

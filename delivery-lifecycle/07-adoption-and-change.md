# 7. 채택과 역할 전환

## 목적

배포된 기능을 공식 업무에 연결하고, 사용자·책임·절차·교육·지원 체계를 바꾼다.

## 주요 작업

1. 실제 사용자가 대표 업무와 예외 업무를 수행하게 한다.
2. 결과 신뢰, 수정 부담, 접근성, 기존 절차 충돌을 관찰한다.
3. 교육·지원·장애 문의 경로를 운영한다.
4. 업무 책임자·운영자·데이터 소유자·승인자 역할을 확정한다.
5. 기존 절차와 새 절차의 이중 운영 기간을 정한다.
6. 기존 절차의 종료·유지·복구 조건을 결정한다.
7. 절감한 시간과 역할 변화를 어디에 재배치할지 합의한다.

## 채택을 보는 관점

| 지표 | 질문 |
|---|---|
| 접근 | 필요한 사람이 실제로 사용할 수 있는가 |
| 반복 사용 | 같은 업무에서 다시 선택하는가 |
| 수정 부담 | 결과를 고치는 시간이 줄었는가 |
| 완료 | 새 흐름으로 업무 결과까지 도달하는가 |
| 신뢰 | 근거와 실패 상태를 이해하고 판단하는가 |
| 업무 영향 | 시간·품질·위험·사업 결과가 바뀌는가 |

## 산출물

- 사용자 검수 기록
- 교육·지원 자료
- 책임 배분표
- 채택·업무 결과 리뷰
- 기존 절차 종료 또는 유지 결정

## 통과 기준

- 나 외의 운영자가 핵심 흐름을 수행할 수 있다.
- 사용하지 않는 이유가 기능·신뢰·교육·절차 중 하나로 구분된다.
- AX Engineer가 모든 일상 운영 요청을 직접 처리하지 않는다.
- 기존 절차의 상태와 최종 책임자가 분명하다.
- 채택 지표와 업무 결과 지표가 분리돼 있다.

## 중단 신호

- 사용자를 교육하지 않고 저항으로 해석한다.
- 로그인 수와 생성 건수만 성과로 보고한다.
- 공식 절차는 그대로인데 새 도구 사용만 권장한다.
- 운영 인수인계 전에 개발 인력이 빠진다.

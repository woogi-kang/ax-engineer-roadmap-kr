# 출처 정책

## 목적

AX Engineer의 역할과 기술 환경은 빠르게 바뀐다. 이 저장소는 기억이나 유행보다 확인 가능한 근거를 우선하고, 사실·경험·가설을 구분한다.

## 출처 유형

| 표기 | 의미 | 사용 예 |
|---|---|---|
| `PRIMARY_OFFICIAL` | 회사·프로젝트·기관의 공식 원문 | 채용 공고, 공식 문서, 표준 |
| `PRIMARY_REPOSITORY` | 공개 저장소의 실제 파일 | README, 기여 규칙, 코드 |
| `PUBLIC_RESEARCH` | 논문·공공 보고서·검증 가능한 연구 | 평가·보안·조직 연구 |
| `PRACTITIONER_EVIDENCE` | 재현 가능한 현장 산출물과 운영 기록 | 평가 결과, runbook, 장애 기록 |
| `PUBLIC_SECONDARY` | 기사·블로그·요약 자료 | 시장 동향과 보조 설명 |
| `HYPOTHESIS` | 아직 검증하지 않은 적용 가설 | 사례의 예상 병목 |
| `UNKNOWN` | 정보가 없거나 상충해 보류 | 조직 책임자, 실제 기준선 |

## 우선순위

1. 직무 정의는 현재 회사의 공식 채용·팀 설명을 우선한다.
2. 기술 동작은 공식 문서와 실제 버전을 확인한다.
3. 공개 프로젝트 평가는 README 요약보다 실제 파일을 읽는다.
4. 기사와 개인 글은 보조 근거로 사용한다.
5. 사례의 성과는 측정 방법·기간·표본이 없으면 수치로 제시하지 않는다.

## 링크 작성

- 링크가 뒷받침하는 주장 가까이에 둔다.
- 변경 가능성이 큰 자료에는 `YYYY-MM-DD 확인`을 표시한다.
- 링크 제목만으로 출처를 평가하지 않는다.
- 접근할 수 없는 링크는 대체 출처를 찾거나 `UNKNOWN`으로 내린다.

## 사례 공개

- 회사·고객·동료를 식별할 수 있는 정보를 제거한다.
- 공개 자료와 비공개 경험을 섞지 않는다.
- 실제 사례를 일반화하면 무엇을 바꿨는지 적는다.
- 데이터 수집 범위와 누락·편향을 함께 공개한다.
- 결과가 없는 실험도 실패 조건과 배운 점이 분명하면 포함할 수 있다.

## 업데이트

출처가 오래됐거나 현재 내용과 다르면 `Source update` Issue를 연다. PR에는 이전 주장, 새 근거, 독자에게 미치는 변화를 적는다.

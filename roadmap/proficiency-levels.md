# 숙련도 기준

## 1. 연차 대신 책임 범위를 본다

AX Engineer 숙련도는 특정 기술을 사용한 기간보다 다음 질문으로 판단한다.

Foundation~Lead는 검증된 산업 표준이나 채용 등급이 아니라, 이 저장소가 다음 학습과 책임 범위를 고르기 위해 제안하는 작업 모델이다.

- 어떤 모호성을 스스로 정리할 수 있는가?
- 어디까지 직접 배포하고 운영할 수 있는가?
- 실패와 위험을 어떤 방식으로 통제하는가?
- 다른 사람과 조직이 결과를 이어받을 수 있는가?
- 한 사례의 학습을 어디까지 재사용할 수 있는가?

## 2. 네 단계

### Foundation

명확하게 제한된 업무를 분석하고 안전한 프로토타입을 검증한다.

- 주어진 문제에서 사용자·입력·출력·성공 기준을 정리한다.
- 공개 또는 비민감 데이터로 얇은 기능을 만든다.
- 정상 사례와 기본 실패 사례를 평가한다.
- 아직 운영 권한과 업무 절차 변경을 단독으로 맡지 않는다.

대표 증거:

- 업무 발굴 카드
- 제한된 프로토타입
- 평가 사례와 결과
- 명시된 가설·제약·비목표

### Builder

한 업무를 기존 시스템과 연결하고 운영 가능한 상태로 배포한다.

- 현업과 범위·우선순위·중단 조건을 합의한다.
- 인증·권한·실패 처리·감사 기록을 포함해 배포한다.
- 사용자 검수와 운영 인수인계를 수행한다.
- 업무 지표와 모델 지표를 나눠 측정한다.

대표 증거:

- 운영 배포와 재현 가능한 코드
- 실행·데이터 계약
- 평가·복구 기록
- 사용자 검수와 인수인계 문서

### Operator

배포된 업무의 품질·비용·안정성·채택을 지속해서 운영한다.

- 장애와 품질 저하를 탐지하고 복구한다.
- 권한·비용·평가 기준의 변경을 관리한다.
- 현업이 직접 조정할 영역과 엔지니어링 영역을 구분한다.
- 기존 수동 절차의 종료 또는 유지 결정을 지원한다.
- 검증된 계약과 구성 요소를 인접한 두 번째 업무에서 재사용하고 호환성을 관리한다.

대표 증거:

- 운영 지표와 SLO
- 장애·복구·회귀 평가 기록
- 채택과 업무 결과 리뷰
- 변경·버전·권한 이력
- 사례 간 재사용과 호환성 기록

### Lead

여러 업무와 팀에서 반복되는 패턴을 조직 역량으로 만든다.

- 업무 포트폴리오와 투자 순서를 관리한다.
- 공통 하네스와 팀별 자율성의 경계를 정한다.
- 보안·데이터·현업·플랫폼 책임자를 연결한다.
- 실패 패턴과 운영 피드백을 공통 기반 로드맵에 반영한다.
- 중앙 팀이 병목이 되지 않는 운영 모델을 만든다.

대표 증거:

- 여러 사례의 비교와 재사용률
- 조직 수준의 완료 판정표
- 공통 실행·평가·감사 기반
- 현업 자립과 역할 전환 기록

## 3. 역량별 기대 수준

| 역량 | Foundation | Builder | Operator | Lead |
|---|---|---|---|---|
| 문제 발견 | 주어진 문제를 구조화 | 현업과 한 업무의 범위 합의 | 운영 중인 업무를 개선·중단 | 조직 포트폴리오와 투자 기준 설계 |
| 업무 설계 | 현재 흐름 작성 | 목표·예외 흐름 구현 | 공식 절차와 역할 전환 | 여러 팀의 공통 규격 |
| 데이터 | 제한된 데이터 계약 | 원본·가공·결과 연결 | 품질·최신성·계보 운영 | 조직 의미 체계와 소유권 |
| 소프트웨어 | 프로토타입과 테스트 | 한 업무의 운영 배포와 통합 | 신뢰성·비용·변경·호환성 운영 | 여러 팀의 재사용 기반과 확장 정책 |
| Applied AI | 기본 평가 | 회귀·안전·사람 인계 | 운영 평가와 품질 개선 | 평가 표준과 모델 전략 |
| 보안·운영 | 위험 식별 | 권한·로그·복구 구현 | 장애·SLO·비용 운영 | 조직 통제 체계 |
| 채택 | 사용자 피드백 | 검수·교육·인수인계 | 채택과 기존 절차 종료 | 자립 가능한 변화 모델 |
| 표준화 | 패턴 후보 기록 | 한 업무 안에서 템플릿화 | 두 번째 업무 재사용과 버전 운영 | 조직 공통 기반 로드맵 반영 |

## 4. 승급이 아닌 성장 판단

다음 수준으로 이동하려면 모든 기술을 새로 배울 필요는 없다. 현재 수준에서 부족한 **책임과 증거**를 하나씩 추가한다.

예시:

```text
Foundation → Builder
프로토타입에 인증·권한·실패 처리·배포·사용자 검수를 추가한다.

Builder → Operator
한 번의 배포를 4주 이상 운영하며 품질·비용·장애·채택 변화를 기록한다.

Operator → Lead
두 개 이상 업무에서 검증한 계약을 여러 팀이 쓰는 기반과 운영 모델로 확장한다.
```

직함이 높은데 증거가 부족할 수도 있고, 직함 없이 Lead 수준의 책임을 맡을 수도 있다. 이 기준은 사람을 평가하기보다 다음 학습과 실습을 선택하는 데 사용한다.
